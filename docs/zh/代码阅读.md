# 代码阅读指南

这份文档给已经读完 `docs/` 概念文档、准备进入源码学习的读者,提供一条**按数据流推进**的阅读路径,而不是逐文件、逐目录地读。它回答"从哪开始、按什么顺序、每个文件解决什么问题"。

> 与本目录其他文档的关系:那些文档讲**概念与用法**(每个子系统是什么、怎么用),这份文档讲**源码阅读顺序**(代码在哪、按什么顺序读、彼此怎么调用)。两者互补——读到某个阶段想深入概念时,再跳到对应的专题文档。

---

## 核心策略

1. **沿一条主数据流自顶向下追**,先把"一个终端从创建到产出"这条线打通,再横向展开子系统。逐文件/逐目录读容易迷失。
2. **以 API 路由为索引**:`api/main.py` 的 40+ 路由就是整个系统的功能目录。想知道某个功能怎么实现,先 `grep` 它的路由,再看它调了哪个 service。
3. **大文件挑函数读,不要通读**:`api/main.py`、`terminal_service.py`、`mcp_server/server.py`、`clients/database.py` 都是"查"的工具,不是"读"的对象(见文末规模速查)。
4. **e2e 测试是最好的地图**:`test/e2e/` 比文档更准确地展示系统**实际**怎么用。

## 阶段 1:主线 —— 一个 agent 终端的生命周期

**目标**:理解"创建终端 → 发输入 → 产出输出"的端到端流程。

```
api/main.py  POST /sessions                 ← 入口(创建 session + 第一个 conductor terminal)
   → services/terminal_service.py  create_terminal()
        → clients/tmux.py            (建 tmux session + pipe-pane)
        → clients/database.py        (terminals / inbox_messages 表)
        → providers/manager.py + providers/base.py + providers/claude_code.py
                                       (provider.initialize():等 shell、发命令、等 IDLE)
        → services/fifo_reader.py     (起 FIFO reader 线程)
api/main.py  POST /terminals/{id}/input   → send_input()
api/main.py  GET  /terminals/{id}/output
```

读完后,`CODEBASE_zh.md` 里"终端创建流程"那张图就全部落地。

> **关键认知**:终端并不在调用方的进程里,而是由 `cao-server` 在后台 **tmux server** 里创建的一个 session。`cao launch` 创建后会执行 `tmux attach-session` 把当前终端"接"上去(见 `cli/commands/launch.py` 与 `backends/tmux_backend.py` 的 `attach_session`)。

**想深入概念**:[`terminal-lifecycle.md`](./terminal-lifecycle.md)(terminal 的创建/累积/清理)、[`tmux.md`](./tmux.md)(直接 attach / list session)。

## 阶段 2:事件驱动管线 —— 项目最精巧的部分

**目标**:理解"输出怎么流动、状态怎么推断、agent 怎么感知彼此空闲"。这是 CAO 区别于"简单 tmux 包装"的核心。本阶段小文件多,适合连续读完:

1. `services/event_bus.py`(109 行)— 先读,事件总线
2. `services/fifo_reader.py`(112 行)— 输出发布者
3. `services/status_monitor.py`(405 行)— **最有价值的一个**:消费 `terminal.{id}.output`,**provider 无关地**推断 `IDLE / PROCESSING / COMPLETED`,再发布 `terminal.{id}.status`。难点在这里:怎么从一堆终端字符流猜出 agent 当前状态
4. `services/inbox_service.py`(175 行)— 消费 status,在 IDLE 时投递排队消息
5. `services/log_writer.py` — 消费 output 写日志,扫一眼即可

管线整体形状:

```
FifoReader ──publish terminal.{id}.output──▶ EVENT BUS ──subscribe──▶ StatusMonitor
                                                  │                        │
                                                  ▼                        ▼
                                             LogWriter             publish terminal.{id}.status
                                                                          │
                                                                          ▼
                                                                   InboxService(IDLE 时投递)
```

读完后,`CODEBASE_zh.md` 的"Inbox 消息流程"图全部落地。

**想深入概念**:[`event-driven-architecture.md`](./event-driven-architecture.md)(整条事件流水线)、[`inbox-delivery.md`](./inbox-delivery.md)(立即投递 + 看门狗兜底两条路径)。

## 阶段 3:多 agent 协作原语

**目标**:理解 `assign` / `handoff` / `send_message` 这些 agent 之间的协作工具。

> ⚠️ **建议把顺序反过来:先读 e2e 测试,再读源码**。带着"它想测什么"的问题去读 `mcp_server` 会顺畅十倍——这些测试就是最好的多 agent 协作教程。

1. **先读测试**:
   - `test/e2e/test_assign.py`
   - `test/e2e/test_handoff.py`
   - `test/e2e/test_send_message.py`
   - `test/e2e/test_supervisor_orchestration.py`(最能体现编排价值)
2. `mcp_server/server.py`(1419 行,大)— **暴露给被编排 agent 用的** MCP 工具(`handoff` / `send_message` / `assign` / `load_skill`)。挑工具读,对照 `mcp_server/models.py`
3. `models/inbox.py` — 消息状态机(`PENDING` / `DELIVERED`)
4. `agent_store/code_supervisor.md` + `developer.md` + `reviewer.md` — supervisor / worker 分工设定
5. `services/flow_service.py` — 作为阶段 1 `create_terminal()` 的一个**调用方**回顾,串起定时任务(cron 轮询 + 条件执行 + 渲染 prompt)

> 注意区分两个入口:`POST /sessions` 新建 session;`POST /sessions/{name}/terminals` 往已有 session 里加 worker(handoff 用)。

**想深入概念**:[`control-planes.md`](./control-planes.md)(Web UI / CLI / MCP / API 四个控制面各自主导什么、流量方向)、[`flows.md`](./flows.md)(定时 flow 的用法)。

## 阶段 4:扩展机制(按兴趣选读)

| 主题 | 文件 | 说明 |
|---|---|---|
| 插件系统 | `plugins/registry.py` + `plugins/base.py` + `plugins/builtin/claude_code_memory.py` + `plugins/events.py` | entry point(`cao.plugins` group)自动发现;`events.py` 是全部事件类型清单,`builtin/` 下是一个完整插件例子 |

## 核心路由速查

`api/main.py` 的路由是整个系统功能的目录。常用的几个:

| 路由 | 用途 |
|---|---|
| `POST /sessions` | 创建 session + 第一个 conductor terminal(`cao launch` 走这里) |
| `GET /sessions` / `GET /sessions/{name}` | 列出 / 查看某个 session |
| `POST /sessions/{name}/terminals` | 往已有 session 加 worker(handoff 用) |
| `POST /terminals/{id}/input` | 给终端发输入 |
| `GET /terminals/{id}/output` | 取终端输出 |
| `POST /terminals/{id}/exit` | 退出终端 |
| `POST /terminals/{receiver_id}/inbox/messages` | 多 agent 通信(send_message) |
| `GET /skills/{name}` | `load_skill` 取 skill 全文 |
| `POST /flows` / `POST /flows/{name}/run` | 定时 flow 管理 |

## 读码技巧

1. **`constants.py` 是配置地图**:所有 `~/.aws/cli-agent-orchestrator/...` 路径(`CAO_HOME_DIR`、`SKILLS_DIR`、`AGENT_CONTEXT_DIR` 等)与默认值都在这里,随时回查。
2. **先扫 `test/e2e/conftest.py`**:它展示怎么起真实 server + 真实 CLI 跑端到端,是理解"运行时拓扑"的最快入口。
3. **CodeGraph(可选但推荐)**:本仓库支持 CodeGraph MCP(`codegraph_*` 工具)。查"谁调用了 X / 改 Z 会影响谁"时,先在仓库根跑 `codegraph init -i` 建索引,之后会比 `grep` 快很多。建不建都行,看你是否常做调用链追踪。
4. **`status_monitor.py` + 各 provider 的提示符正则是精华**:CAO 最 hard-core 的部分是用字符流解析"agent 现在忙不忙"。`claude_code.py` 开头那堆 regex 和 `status_monitor.py` 的状态机值得反复看。
